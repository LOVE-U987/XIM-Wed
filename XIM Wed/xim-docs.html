XIM Studio 代码协议说明文档

最后更新: 2025-11-15 版本: v1.2

1. XIM Studio 代码协议

XIM Studio 代码（XIM Studio-code）用于在本工作室生态内区分「信任用户」与「普通成员」身份，并据此授予对应资源的访问权限

1.1 协议目的

XIM Studio 代码（XIM Studio-code）用于在本工作室生态内区分「信任用户」与「普通成员」身份，并据此授予对应资源的访问权限

鉴于工作室已公开多数代码仓库，惟处于开发或待验证阶段之项目尚不成熟，故部分资源仅向持有 XIM Studio 代码之内部人员开放

1.2 代码类型与权限矩阵

权限对比表

代码类型 适用人群 权限范围 可使用时长 备注
访问码 外部测试者 开发预览页、公开文件、已开放项目 永久有效 —
信任码 信任用户 内部文件、非公开分支、灰度功能 短期有效 须绑定访问码后方可申请
工作码 核心开发组 全部内部资源、生产环境、CI/CD 流水线 永久有效 禁止外泄；一经发现即刻作废并追责

1.2.1 访问码 (Access Code)

权限范围：开发预览页、公开文件、已开放项目

适用人群：外部测试者

可使用时长：永久有效

1.2.2 信任码 (Trust Code)

权限范围：内部文件、非公开分支、灰度功能

适用人群：信任用户

可使用时长：短期有效

备注：须绑定访问码后方可申请

1.2.3 工作码 (Work Code)

权限范围：全部内部资源、生产环境、CI/CD 流水线

适用人群：核心开发组

可使用时长：永久有效

备注：禁止外泄；一经发现，XIM Studio 有权立即停用并追究相应责任

1.3 代码格式规范

  • 访问码：小写英文单词 + 四位数字，示例 teacher1021
  • 信任码：大写字母 + 四至五位数字，示例 CGW-1077
  • 工作码：格式同信任码，惟系统内部标记为 internal 级别

1.4 使用周期与失效规则

  1. 访问码对应之小范围测试项目结束后，项目级限制将自动解除，原代码仍保留查看权限
  2. 信任码与工作码于分发时，将明确告知「有效期」及「可访问仓库列表」
  3. 任何代码一经触发「恶意分享」检测（包括但不限于仓库日志异常、下载 IP 异常等），将被立即吊销，恕不另行通知

1.5 获取与申请流程

申请条件与处理时效

申请对象 必要条件 平均处理时效
访问码 说明用途即可 1–2 工作日
信任码 ① 已持有访问码 ② 说明具体用途 2–3 工作日
工作码 ① 内部 HR 备案 ② 组长实名担保 3–5 工作日

申请须知

若 4 个工作日未获回复，可再次提交申请或直接致信 Studio Lead：Love_U2025@outlook.com

1.6 分享与合规条款

  • 访问码：允许公开分享，但禁止用于刷单、爬取等非测试目的；
  • 信任码：仅限申请人及其直属团队使用，不得二次分发；
  • 工作码：绝对禁止外泄；一经发现，XIM Studio 有权立即停用并追究相应责任

1.7 联系渠道

  • 邮件（推荐）：studio@shaole.me
  • 紧急通道：组长或项目架构师（仅对内部有效）

1.8 代码安全与保护措施

代码加密机制

  • 传输加密：所有代码传输采用SSL/TLS 1.3加密协议，确保数据在传输过程中不被窃取或篡改
  • 存储加密：代码在服务器端存储时采用AES-256加密算法，密钥定期轮换
  • 本地缓存加密：客户端本地缓存的代码片段采用设备绑定的加密密钥保护

访问控制机制

  • 最小权限原则：仅授予用户完成任务所需的最小权限
  • 多因素认证：获取信任码和工作码时需启用多因素认证
  • IP地址白名单：支持设置可信IP地址范围，限制代码访问来源
  • 会话管理：定期刷新会话令牌，超时自动登出

安全审计与监控

  • 访问日志：记录所有代码访问行为，包括访问时间、IP地址、操作类型等
  • 异常检测：实时监控异常访问模式，如频繁失败尝试、异常下载量等
  • 定期审计：每季度进行一次全面的安全审计，评估代码访问控制有效性
  • 漏洞扫描：定期对代码仓库进行漏洞扫描，及时修复安全隐患

1.9 协议更新与修订

版本管理

  • 版本号格式：采用主版本号.次版本号.修订号格式（如v1.0.0）
  • 主版本更新：当协议内容发生重大变更，影响用户权利义务时，主版本号递增
  • 次版本更新：当协议内容发生非重大变更，如新增功能或优化流程时，次版本号递增
  • 修订号更新：当仅对协议进行文字性修正或轻微调整时，修订号递增

更新通知

  • 通知方式：协议更新后，将通过平台公告、邮件通知、站内信等方式告知用户
  • 通知期限：重大更新将提前30天通知用户，非重大更新将在更新后7天内通知
  • 生效时间：更新后的协议自通知中指定的生效日期起生效

修订流程

  1. XIM Studio团队根据业务发展、法律法规变化或用户反馈，提出协议修订需求
  2. 对修订内容进行内部审核和法律合规性评估
  3. 发布修订草案，征求用户意见（重大修订）
  4. 根据用户反馈进行调整和完善
  5. 正式发布更新后的协议，并通知所有用户

用户权利

用户有权选择是否接受更新后的协议如用户不接受，应停止使用XIM Studio平台及相关服务；如用户继续使用，则视为接受更新后的协议

1.10 法律适用与争议解决

法律适用

本协议的订立、履行、解释及争议解决均适用中华人民共和国法律（不包括香港特别行政区、澳门特别行政区和台湾地区法律）

争议解决机制

  1. 协商解决：如双方发生争议，应首先通过友好协商的方式解决
  2. 调解解决：协商不成的，可向有关调解机构申请调解
  3. 仲裁解决：调解不成的，任何一方均有权将争议提交至上海仲裁委员会，按照该会届时有效的仲裁规则进行仲裁
  4. 诉讼解决：如仲裁条款无效或无法执行，任何一方均有权向有管辖权的人民法院提起诉讼

管辖法院

如涉及诉讼，双方同意将争议提交至XIM Studio住所地有管辖权的人民法院管辖

其他规定

  • 本协议的任何条款被认定为无效或不可执行，不影响其他条款的效力
  • 双方均应遵守本协议的各项约定，任何一方违反本协议约定给对方造成损失的，应承担相应的赔偿责任

2. [MOR]Major overhaul and reorganization

最后更新: 2025-11-30 版本: v1.0

2.1 项目背景

这是一"坨"给偏技术玩家准备的高画质客户端整合包AI 测评给了 7/10，毒舌点在于："把它当高画质启动器而非即开即玩包，就不会踩坑"

官网已上线：https://modrinth.com/modpack/mor

2.2 项目目标

打造一个视觉效果出色的高画质客户端整合包，为偏技术玩家提供个性化的游戏体验

2.3 功能特点

核心功能特点

  • 高画质视觉效果：视觉完成度 4/5，截图即壁纸，为玩家提供沉浸式的游戏视觉体验
  • 性能表现：不是极端 FPS 向，是特效向1060+8G 以下别来，核显直接 30 FPS 献祭
  • 丰富的功能键：30+ 功能键，提供丰富的操作选项，满足技术玩家的个性化需求
  • 中文支持：提供中文汉化资源包（MOR-Setting），但必须版本完全对应，否则可能报错

2.4 版本信息

版本详情

  • 最新版本：2.1.3
  • 许可证：MIT
  • 格式：.mrpack 格式，拖进启动器就能跑
  • 注意事项：别扔进服务端，纯客户端整合，服务器会炸

2.5 上手难度

难度说明

  • 难度评级：2/5 星
  • 特点：30+ 功能键，配置界面全英文，新手 10 分钟迷路
  • 汉化方式：拖入资源包文件夹 → 置顶加载 → 界面变中文
  • 更新方式：整包替换，配置需重调，无自动备份所以别瞎调配置，调了记得自己备份

2.6 中文支持

中文资源

  • BBSMC 镜像站：https://bbsmc.net/modpack/mor（更新比 Modrinth 慢）
  • B站教程：有持续更新的设置教程（搜MOR），键位、光影、低配救帧都在
  • 问题反馈：去BBSMC，我们集中处理

2.7 团队评价

评价详情

  • 视觉完成度：4/5
  • 性能：3/5（好看但吃配置）
  • 上手难度：2/5（30 个按键魔鬼训练营）
  • 整体评分：7/10，对得起"大修与重组织"这个名字

一句话总结：想开箱即用？快跑想当高画质启动器自己折腾？欢迎

创建一个灵活的Minecraft整合包，作为我们的试验田，测试新模组、新点子，同时提供一个有趣的游戏体验

6.3 功能特点

核心功能特点

  • 多样化玩法：支持科技、魔法、生存等多种玩法，玩法跟着兴趣走，今天想肝科技，明天可能就去养蜜蜂了
  • 试验田功能：所有新模组、新点子、新 bug 都在这里首发，作为我们的技术试验田
  • 共同存档：不只是测试工具，更是我们这群人的共同存档，记录我们的游戏历程
  • 多版本支持：支持Minecraft 1.20.X–1.21.X，兼容多个游戏版本，满足不同玩家的需求
